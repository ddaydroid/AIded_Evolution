### Task 1: Intelligent tool output truncation for large results
Files: src/main.rs, src/format.rs
Description: When bash commands or read_file return very large output (>30,000 chars), the full content goes into the context window, burning tokens fast and degrading model performance. Claude Code truncates intelligently — keeping the first N and last N lines with a "[truncated X lines]" marker in between. Implement a `truncate_tool_output(output: &str, max_chars: usize) -> String` function in format.rs that preserves first ~100 lines and last ~50 lines when output exceeds the threshold, with a clear truncation marker. Apply this in the tool execution wrappers (GuardedTool and ConfirmTool execute methods) and in the BashTool confirm wrapper — the truncation should happen on the ToolResult content before it goes back to the model. Add tests for: output under threshold (unchanged), output over threshold (truncated with marker), edge cases (empty, exactly at threshold). This is the single biggest practical gap for real-world use — a developer running `cat` on a large file or `cargo test` with verbose output currently floods the context.

### Task 3: `/retry` with error context — smarter error recovery
Files: src/commands.rs, src/prompt.rs
Description: Currently `/retry` just re-sends the last user input verbatim. When a tool call fails (e.g., edit_file can't find the target text, bash command exits non-zero), Claude Code's model automatically gets the error and tries a different approach. yoyo already does this via the normal agent loop — but `/retry` doesn't carry forward any error context from the failed attempt. Enhance `/retry` to optionally include a brief error summary: when the last agent turn ended with a tool error or non-zero exit, prepend a note like "[Previous attempt failed: {error_summary}. Try a different approach.]" to the retried prompt. This helps the model avoid repeating the same mistake. Add the error tracking to `run_prompt` (store last error in a new `last_error: Option<String>` field on SessionChanges or a separate tracker), and use it in `handle_retry`. Write tests for: retry with no error (unchanged behavior), retry with error context (prepended note), error cleared after successful turn.
Issue: none
